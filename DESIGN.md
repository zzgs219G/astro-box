## 核心设计哲学

### 内容至上（Content First）
* 界面永远为内容服务，绝不抢戏。
* 大量留白（Generous Whitespace）让文字和图片自然呼吸。
* 所有装饰性元素都被极致简化，视觉焦点始终集中在内容本身。

### 丝滑的流动感（Fluid & Buttery Smooth）
* 所有交互（hover、点击、滚动、主题切换）都追求高级、轻盈、不卡顿的手感。
* 动画遵循「慢进快出」的自然规律，避免生硬的弹跳或过度效果。

### 现代极简主义 + 柔和日系美学
* 以柔和暗色模式为主，圆角、阴影、间距都经过精心调校，营造高级感和舒适度，而非冰冷的工业风。
* 排版极致讲究：舒适的行高、清晰的层级、优雅的标题字体搭配。

### 隐形设计（Invisible Design）
* 好的 UI 应该是「感觉不到它的存在」。导航栏、侧边栏、搜索等功能采用轻量化设计，在需要时出现，不需要时几乎隐形。
* 避免视觉噪音（No Visual Noise）：无过多分割线、无多余图标、无花哨特效。

## 设计原则细节

* **颜色系统**：基于 OKLCH 色彩空间 + CSS 变量 `--hue`，实现单色调优雅主题，支持亮/暗模式无缝切换。
* **排版系统**：正文追求极致阅读舒适度，代码使用 JetBrains Mono，标题采用柔和的层级对比。
* **交互反馈**：所有可点击元素都有细腻的 `hover` 和 `active` 状态，使用统一的平滑呼吸感。
* **响应式**：从手机到大屏都保持一致的高级感，采用窄栏居中阅读布局。

Fuwari 的 UI 不是为了炫技，而是为了让作者的内容「安静而优雅地发光」。它追求的是：打开即舒适、阅读即享受、浏览即愉悦的高级沉浸式体验。

## 三、动画系统（最关键约束）

### 3.1 允许的动画（完整白名单）

只有以下两种，没有第三种：

```
✅ opacity 渐变（fade in / fade out）
✅ translateY 位移，且绝对值 ≤ 8px
```

两者必须在**容器级**使用，不得应用于单个子元素。

### 3.2 禁止的动画（完整黑名单）

```
❌ scale（任何形式的缩放）
❌ translateX（横向位移）
❌ translateY > 8px（过大的垂直位移）
❌ rotate（旋转）
❌ 弹簧/回弹/果冻/过冲（spring、bounce、elastic、overshoot）
❌ 交错动画（stagger，即对列表每项设置不同 delay）
❌ FLIP 动画（即基于位置差值的位移补间）
❌ 卡片悬浮升起（box-shadow 骤变 + translateY 同时触发）
❌ 数学公式区域的任何动效（必须完全静止）
```

### 3.3 动画层级规则

动画**只允许**作用于以下层级：

```
✅ Page    — 整页进入/离开
✅ Section — 内容区块整体替换
✅ Container — 卡片网格容器整体
```

动画**严禁**作用于以下层级：

```
❌ Card（单张卡片）
❌ Button（单个按钮）
❌ Tag（单个标签）
❌ List Item（单条列表项）
❌ Icon（图标）
```

### 3.4 一句话原则

> 动画只表达"状态已切换完成"，不表达"元素正在运动"。

## 四、交互模型

### 4.1 列表/网格更新（三步走，不可简化也不可复杂化）

当用户切换分类或标签时：

```
步骤 1：Grid 容器整体 fade out（透明度 1 → 0）
步骤 2：数据源完成过滤，替换内部内容
步骤 3：Grid 容器整体 fade in（透明度 0 → 1）
```

**绝对禁止：**
- 单张卡片独立入场（每张卡片各自 fade in）
- FLIP 位移（卡片从旧位置滑动到新位置）
- 增量更新（只动"变化的"卡片）

### 4.2 搜索框防抖特例（唯一例外）

`searchQuery`（用户在搜索框打字）触发的列表更新：

```
✅ 网格直接瞬间刷新，无任何过渡动画
❌ 禁止触发 fade 流程（会导致每次击键闪烁）
```

### 4.3 Hover 反馈（允许的完整列表）

```
✅ opacity 微调（如 hover:opacity-80）
✅ 背景色轻微偏移（如 hover:bg-black/5）
✅ 文字颜色柔和过渡（transition-colors，duration ≤ 150ms）
```

```
❌ scale 放大
❌ translateY 上移（卡片"浮起"）
❌ box-shadow 骤增（"立体感"）
❌ 多元素联动 hover（hover 一个元素，另一个元素也动）
```

## 五、布局系统

### 5.1 Grid 稳定性

卡片网格的外层容器必须指定 `min-h`，防止内容少时布局塌陷：

```html
<!-- 正确 -->
<div class="grid min-h-[500px] ...">

<!-- 错误：无最小高度，内容少时页面抖动 -->
<div class="grid ...">
```

可使用 `content-visibility: auto` 减少 reflow，但不强制要求。

### 5.2 空状态

搜索或筛选无结果时：

```
✅ 空提示组件继承 Grid 容器原有尺寸（不塌陷）
✅ 居中显示，文字极简
❌ 不允许用图片、插图、动效装饰空状态
```

## 六、色彩系统

### 6.1 基本规则

全站色彩基于 CSS 变量 `--hue` + **OKLCH 色彩空间**延伸，不允许偏离。

### 6.2 禁止的写法

```css
/* ❌ 硬编码色值 */
color: #ffffff;
background: #000;

/* ❌ 无透明度修饰的强硬类名 */
bg-white
bg-black
text-white
text-black
```

### 6.3 正确的写法

```html
<!-- ✅ 带透明度的语义类 -->
<div class="bg-white/60 dark:bg-[#18181b]/60 backdrop-blur-sm">

<!-- ✅ 基于 oklch 变量推导 -->
<div style="background: oklch(var(--card-bg))">
```

### 6.4 暗黑模式

所有背景色和文字色必须同时声明亮色和暗色版本。不允许只写亮色版本然后"暗色模式自动处理"。

## 七、排版

### 7.1 字体分层

| 层级 | 要求 |
|------|------|
| 标题 | 高对比度，低装饰性 |
| 正文 | 舒适行高（line-height ≥ 1.7），优先可读性 |
| 代码 | 全站锁死 **JetBrains Mono**，不允许替换 |

### 7.2 分割线

- 全站**不使用硬编码装饰性边界线**（`border`、`<hr>` 等）
- 内容分区靠**留白**实现，不靠线条

---

## 八、Swup 页面过渡

```
✅ 只用于页面级（Page）切换
✅ 转场效果只允许 fade（透明度渐变）
❌ 不允许用于组件内部状态切换
❌ 不允许用于控制卡片显隐
❌ 不允许添加位移、滑动、翻转等转场效果
```

## 九、组件职责边界

### ResourceCard.svelte

**只做两件事：**
1. 渲染静态内容
2. 展示基本状态（如标签、日期）

**禁止承载：**
- 任何动画逻辑
- 跨组件状态切换
- 自身的过滤/排序逻辑

### ResourceExplorer.svelte

**唯一的状态管理中心，负责：**
1. 数据源过滤
2. 容器级整体过渡
3. 大类 + 子标签的联动状态

## 十、AI 执行时的自检清单

每次生成或修改代码前，逐项检查：

```
□ 是否为单个卡片/按钮/标签添加了独立动画？→ 删除
□ 是否使用了 scale、translateX、bounce？→ 删除
□ 是否引入了新的动画库或 JS 动画逻辑？→ 删除
□ 是否硬编码了色值（#fff、#000 等）？→ 替换为变量
□ 是否只声明了亮色模式而忘记暗色？→ 补充
□ 是否用 Swup 控制了局部组件？→ 拆分职责
□ 搜索输入时是否触发了 fade 动画？→ 改为直接刷新
□ Grid 容器是否缺少 min-h？→ 补充
□ 空状态是否导致布局塌陷？→ 修复
```

## 十一、一句话判断法则

> 如果你认为"这个功能需要让某个元素动起来才显得合理"，
> 那么**不是实现方式错了，是设计方向本身就错了**。
> 回到"整体容器状态替换"模型重新设计。
